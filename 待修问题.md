## P0 — 模式② 直写恢复时 seek 到段开头而非 confirmed_bytes

**位置**：`chunk_assembler.py:848-850` `_assemble_with_sequential_write`

**根因**：模式② 续传时，`f.seek(output_offset)` 总是定位到段开头 `seg.start`：

```python
# line 848-850
if output_offset > 0:
    f.seek(output_offset)
```

如果 `start_term_idx > 0`（之前已写入部分 term），此处应该定位到**已写入数据之后**而非段开头。不修复的话，续传时新写入的数据会覆盖段开头已有数据。

`confirmed_bytes` 已经在 line 699-702 从 checkpoint 加载（记录的是最后保存时的绝对文件位置），但没有用于 seek。

**合并说明**：原"P1 — confirmed_bytes 与 output_offset 语义不一致"与本次同根因，已合并。

**方案**：将 seek 逻辑改为：
```python
if output_offset > 0:
    seek_pos = output_offset
    if start_term_idx > 0:
        if confirmed_bytes > 0:
            seek_pos = confirmed_bytes
        else:
            for i in range(start_term_idx):
                if i == 0:
                    seek_pos += max(0, recon.terms[i].unpacked_length - recon.offset_into_first_range)
                else:
                    seek_pos += recon.terms[i].unpacked_length
    f.seek(seek_pos)
```

## P1 — Worker `queue.put` 在 writer 崩溃后可无限阻塞

**位置**：`segmented_reconstructor.py:691` `_process_segment_parallel`

**根因**：`self._write_queue.put(...)` 不带超时（blocking indefinitely）。如果 writer 线程因磁盘 I/O 错误崩溃（line 643-647），队列不再消费。Worker 填满 `maxsize` 后 `put()` 永久阻塞 → 所有 future 卡住 → main thread 的 `as_completed` 永不返回 → 死锁。

已有 `_writer_exception_event` 检测，但只在有 future 完成时触发。如果所有 worker 都在阻塞等待 queue.put，没有任何 future 能完成。

**当前风险**：`maxsize = parallel_segments * 2`（line 493），worker 数 ≤ `parallel_segments`，每个 worker 只 put 1 次 → 队列不会满。仅在异常重试路径下有理论风险。

**方案**：`queue.put(..., timeout=5.0)` 配合 `self._stop_event.is_set()` 兜底：
```python
while True:
    try:
        self._write_queue.put((seg.start, seg.index, seg_temp, seg_size), timeout=5.0)
        break
    except queue.Full:
        if self._stop_event.is_set():
            seg_temp.unlink(missing_ok=True)
            return
        continue
```

## P1 — `_shared_refresh_recon` 无 TTL 清理

**位置**：`cas_client.py:99-100` + `541-555`

**根因**：`_shared_refresh_recon` 写入时打上了时间戳（line 664: `time.time()`），消费时检查了 120 秒 TTL（line 544）和 60 秒 TTL（line 693）。**但这个对象本身从不会被清理**——即使已经过期，它仍然驻留在内存中。

当 file_hash 不匹配时（换文件下载），旧结果虽然被 TTL 过滤跳过，但占用了内存（一个 QueryReconstructionResponse 可能几 MB）。更重要的是，如果同一个 CASClient 实例被用于下载不同文件（`file_hash` 不同），旧的共享刷新结果虽然不会被使用（TTL + file_hash 双重检查），但永远不会被 GC。

**方案**：在写入 `_shared_refresh_recon` 时增加过期检查：
```python
with self._shared_refresh_lock:
    if self._shared_refresh_recon is not None:
        _, _, old_ts = self._shared_refresh_recon
        if time.time() - old_ts > 120:
            self._shared_refresh_recon = None
    self._shared_refresh_recon = (file_hash, recon, time.time())
```

## P1 — 低速续传 Range 调整后 attempt 仍递增

**位置**：`cas_client.py:719-741`

**根因**：`LowSpeedTimeoutError` 的 handler 调整了 Range（line 733-734: `new_start = current_range.start + e.received`），然后用 `continue` 跳回 for 循环顶部。**但 `attempt` 并不递减**——下次循环 `attempt` 已经 +1 了。

注释说"低速超时不算入重试次数（只调整范围）"，但代码实际上 **仍然消耗了 attempt**。如果一个 xorb 反复低速续传 5 次，就会触发 `raise RuntimeError("已重试 5 次")`，即使每次都成功接收了部分数据。

**影响**：在网络慢但持续的情况下，可能过早放弃下载。原始 retry_max=5 次低速续传 ≈ 5×30s = 150s 的容忍窗口，但 xorb 的下载可能需要多轮低速续传才能完成。

**方案**：在 `LowSpeedTimeoutError` handler 中递减 attempt：
```python
except LowSpeedTimeoutError as e:
    # ...
    # 低速超时不算入重试次数（只调整范围）
    attempt -= 1  # 抵消 for 循环的 +1
    continue
```

## P2 — 顺序写入最后一批 term 无 final fsync

**位置**：`chunk_assembler.py:960-964`

**根因**：每 `fsync_interval=5` 个 term 做一次 fsync，但退出 `with open` 后没有 FINAL fsync。Python 的 `f.close()` 只做 `flush()`（写到内核缓存），不做 `os.fsync()`（刷到物理存储）。最后不足 5 个 term 的数据可能在 OS 缓存中。

进程级崩溃是安全的（close 的 flush 已到内核），但 OS 级崩溃（掉电）可能丢失最后几个 term。

**方案**：在 `with` 块退出前补一次 final fsync：
```python
# 循环结束后、with 退出前
f.flush()
os.fsync(f.fileno())
```

## P2 — recon 缓存并发写

**位置**：`segmented_reconstructor.py:817-835` `_save_recon_cache`

**根因**：`_save_recon_cache` 直接 `open(cache_path, 'w')` 写入——没有锁保护。在并行模式下（parallel_segments > 1），多个 worker 可能同时对不同 segment 进行 _process_segment_parallel，但每个 segment 的缓存文件名不同（`seg_N.recon.json`），所以即使并发写也不会冲突。

**但**：如果 `_load_recon_cache` 和 `_save_recon_cache` 被同一个段重入（理论上不会，但如果 future 取消+重试可能发生），则可能出现读-写竞争。

**影响**：极低概率，仅在异常重试路径下。

**方案**（可选）：给 `_save_recon_cache` 和 `_load_recon_cache` 加 `self._lock` 保护。

## P2 — checkpoint `_save_unsafe` 非原子写入

**位置**：`checkpoint_manager.py:289-308`

**根因**：`_save_unsafe` 直接 `open(self.checkpoint_path, 'w')` 写入 JSON。如果进程在写入中途崩溃（例如 kill -9），checkpoint 文件会处于截断/损坏状态。下次加载时 `json.load()` 抛出 `JSONDecodeError`，导致 **整个 checkpoint 丢失**——包括之前所有已保存的 term 进度和 per-term SHA256。

对比 `SegmentCheckpointManager.save()`（`segmented_reconstructor.py:117-149`）使用了 tmp + replace 的原子写入模式。

**影响**：在 checkpoint 文件较大的场景（数千个 term 的 per_term_hashes）下，写入时间可能达到几十毫秒，崩溃窗口不小。

**方案**：改为 tmp + replace 原子写入：
```python
def _save_unsafe(self, checkpoint):
    if not self.checkpoint_path:
        return
    try:
        self.checkpoint_path.parent.mkdir(parents=True, exist_ok=True)
        data = checkpoint.to_dict()
        tmp_path = self.checkpoint_path.with_suffix('.tmp')
        with open(tmp_path, 'w', encoding='utf-8') as f:
            json.dump(data, f, indent=2)
            f.flush()
            os.fsync(f.fileno())
        tmp_path.replace(self.checkpoint_path)
        self._cache = checkpoint
    except Exception as e:
        logger.error(f"[Checkpoint] 保存失败: {e}")
```

## P2 — ACC release 不配对 acquire 时信号量计数溢出

**位置**：`adaptive_concurrency.py:112-114,199-200,226-229`

**根因**：`release()` 直接调用 `self._semaphore.release()`，不检查是否有对应的 `acquire()`。在 `_maybe_increase` 中（line 199-200），增加并发度时调用 `self._semaphore.release()` 增加一个许可——但这个许可没有对应的 acquire。

如果并发度先增后降（increase → decrease），减降时只调整 `_current` 计数不做 `acquire`（line 227 注释："不能调用 acquire，因为可能阻塞"）。这导致**信号量实际许可数 > _current**。

短期内这不是致命问题（多出的许可会让下次 acquire 更快通过），但长期累积后信号量的 permit 数会漂移到一个比预期更高的值，**失去限流效果**。

**影响**：中等。在高并发反复升降的场景下，ACC 可能无法有效限制并发数。

**方案**：在 `_maybe_decrease` 中记录"待回收"的许可数，在下次 release 时扣减：
```python
def _maybe_decrease(self, reason):
    with self._lock:
        if self._current <= self._min:
            return
        self._current -= 1
        self._pending_acquire = True  # 标记需要扣减

def release(self):
    with self._lock:
        if self._pending_acquire:
            self._pending_acquire = False
        else:
            self._semaphore.release()
```

## P2 — 顺序写入 truncate 前无断言保护

**位置**：`chunk_assembler.py:861`

**根因**：当 `need_truncate` 为 True 时 `f.truncate(bytes_already_written)` 可能截断到比 `existing_size` 更短的位置。在顺序写入模式下，term N 损坏 → N 之后的数据不可信，截断是安全的。但应加断言保护。

**方案**：
```python
assert bytes_already_written <= existing_size, f"truncate will lose data: {bytes_already_written} > {existing_size}"
f.truncate(bytes_already_written)
```

## P3 — 并行段各 tracker 的 xorb/term 总数在进度中弹跳

**位置**：`segmented_reconstructor.py:715-749` `_assemble_segment_data`

**根因**：每个 worker 独立创建 ProgressTracker，各自的 `total_xorbs`/`total_terms` 是 per-segment 口径（段 A 有 45 xorbs，段 B 有 50 xorbs）。白名单过滤后这些值仍会通过回调到达 `_latest_stats`，导致显示在 `8/45` 和 `12/50` 之间弹跳。

**影响**：仅影响中断/异常时的统计显示（`download.py:868-884`），不影响 writer 上报的 `assembled_bytes`/`completed_segments` 等关键字段。

**方案**（可选）：worker 回调中不传递 `total_xorbs`/`total_terms`/`current_segment`，只在 writer 或全局统一设置。

## P3 — download.py segmented 路径 xorb_hashes 未初始化

**位置**：`download.py:809,857`

**根因**：segmented 路径（line 809）：
```python
result_path = reconstructor.reconstruct_file(resume=args.resume)
```
只返回 `result_path`（Path）。而 line 857 引用了 `xorb_hashes`：
```python
if chunk_cache and chunk_cache.enabled and xorb_hashes:
```
`xorb_hashes` 只在非 segmented 路径（line 836）中通过解包赋值：
```python
result_path, xorb_hashes = reconstructor.reconstruct_file(...)
```
segmented 路径下 `xorb_hashes` 从未被赋值 → **NameError**。

**影响**：仅影响 segmented 模式 + chunk_cache 启用 + 下载成功后的清理逻辑。不致命但会导致异常退出（已成功下载的文件不受影响）。

**方案**：将 `xorb_hashes` 初始化为空列表：
```python
xorb_hashes = []
if not use_segmented:
    result_path, xorb_hashes = reconstructor.reconstruct_file(...)
```
