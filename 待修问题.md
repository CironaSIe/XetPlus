## P0 — CAS 预刷新走缓存拿相同 URL，形同虚设

**位置**：`cas_client.py:606-618` `get_xorb_data_with_retry()`

**根因**：pre-refresh 调 `get_reconstruction()`，其内部先查 `_recon_cache`（30s TTL）。如果缓存未过期，直接返回——**URL 和上一次一模一样**。pre-refresh 看起来"刷新"了（日志打"预检刷新 URL"），实际 `current_url` 同值赋值，毫无作用。

**循环**：
```
t=0:   pre-refresh → cache miss → API → 新 URL → 填cache
t=30:  pre-refresh → cache HIT  → 同URL → 下载 → (也许) 403
       403 handler → pop cache → API → 真新 URL → 填cache
t=60:  pre-refresh → cache HIT  → 同URL（这次不403了）
```

**方案**：
1. pre-refresh 也先 pop cache，确保拿到真新 URL
2. 或完全移除 pre-refresh，403 handler 承担全部刷新职责

## P0 — pre-refresh 异常吞没，留旧 URL 继续用 → 403

**位置**：`cas_client.py:617-618`

```python
except Exception:
    pass  # 吞掉！current_url 还是旧的
```

`get_reconstruction()` 超时或 `_find_xorb_in_recon` 抛异常时，异常静默吞掉，`current_url` 不更新。下载直接用原始 URL → 403。

**方案**：至少打 warning 日志。或统一走 403 handler 刷新路径。

## P0 — 每次续传丢弃最后一个未确认的分片

**位置**：`segmented_reconstructor.py:637-640`

**根因**：`_process_segment_parallel` 中数据入队和标记完成不是原子的：

```python
# line 637-640
self._write_queue.put((seg.start, segment_data))  # ① 数据入队
self._mark_segment_completed(seg)                  # ② 标记完成 ← 可能来不及执行
```

如果进程在 ① 和 ② 之间崩溃：
- 数据可能已被 writer 线程写入输出文件 ✅
- 但 checkpoint 没标记该段完成 ❌
- 续传时 checkpoint 说"没做" → 重新下载 → 该段被"丢弃"

**同样的 gap 存在于模式 ②**：`_process_segment` 中 `shutil.copyfileobj` 完成后到 `_mark_segment_completed` 之间。

**方案**：`_mark_segment_completed` 移到 writer 线程，在 **fsync 之后**立即调用。确保 checkpoint 里的段 = 已落盘的段。

## P1 — 并行段各 ProgressTracker 独立上报，竞态覆盖

**位置**：`segmented_reconstructor.py:652-680`、`download.py:779-786`

**根因**：并行模式下每个段创建一个 `ProgressTracker`，各持口径不同的统计（per-segment 的 xorb/term 数），全部通过同一个回调写入同一个 `_latest_stats` 字典：

```
段0 Tracker → _worker_cb 过滤 → self.progress_callback → _latest_stats.update()
段1 Tracker → _worker_cb 过滤 → self.progress_callback → _latest_stats.update()  ← 覆盖
writer 线程 → self.progress_callback → _latest_stats.update()                     ← 覆盖
```

**表现**：
- xorb 进度在 `8/45` 和 `12/50` 之间反复跳（段口径不同）
- `total_segments` 被不含该字段的回调覆盖掉
- `completed_segments` 滞后 1（writer 读时 mark_completed 还没执行）
- `current_segment` 乱序跳变（段完成顺序不确定）

**方案**：writer 线程统一负责写入进度上报（bytes + segments），worker 不报进度。per-segment xorb/term 下载进度改用累加器（文件级合计），不参与 `_latest_stats` 竞争。

# 附：设计修改评估

## 修改方案：mark_completed 移到 writer 线程

**原则**：写入成功（fsync）后，才标记完成。

```
Worker:   下载 → 解压 → put(start, data, seg_index)          → done
Writer:   get → seek → write → fsync → mark_completed(seg_index) → callback
                                                         ↓
                                              checkpoint 保存 + 上报进度
```

## 积极面

| 维度 | 效果 |
|------|------|
| 续传可靠性 | ✅ checkpoint 中的段 = 已 fsync 的段，间隙消除 |
| 进度准确度 | ✅ bytes 和 segments 由唯一源头（writer）上报，无竞态 |
| confirmed_bytes | ✅ 直接来自 writer 的已写入量，不再用段内逻辑偏移 |
| 分段完成顺序 | ✅ 无关，checkpoint 用 set 存储 (start,end)，不受顺序影响 |
| 模式 ② | ✅ 不依赖 writer 线程，但同样原则——copyfileobj 后加 fsync 再 mark_completed |

## 隐患与风险

### ① Writer 线程成为吞吐瓶颈

Worker 解压后的数据通过 `put()` 入队。如果磁盘写入速度 < 网络下载 + 解压速度，队列填满后 worker 的 `put()` 阻塞 → 下载停滞。

**当前缓解**：队列大小 `maxsize=parallel_segments*2`（如 4 路并行 = 8 槽）。如果每段 256MB，队列缓冲上限 = 2GB 内存。

**风险放大**：加上 `mark_completed` 内的 checkpoint I/O（JSON 序列化 + 磁盘写入 + 原子替换），writer 处理单段数据的耗时增加，队列满的窗口更大。

**建议**：checkpoint 保存放到独立线程，writer 只负责写数据和发"完成"信号。

### ② Writer 线程崩溃不被上层感知

当前 `_writer_worker` 异常时（line 599-601）只 `logger.error` + `raise`，但 daemon 线程的异常不会传播到主线程：

```python
# 主线程
self._write_queue.put(None)         # 往队列放结束标记
self._writer_thread.join()          # 已崩的线程立即返回
# → 主线程以为一切正常
```

如果修改后 writer 承担了 mark_completed 职责，writer 崩溃意味着 checkpoint 丢失、进度停滞，但主线程不知道。

**建议**：writer 异常通过 `threading.Event` 或异常变量传播到主线程。

### ③ Segment 粒度感知

Writer 当前只处理 `(offset, data)` 元组，不知道属于哪个 segment。要调 `_mark_segment_completed(seg)` 需要知道 `seg_index` 或 `SegmentInfo`。

**需改**：队列元组改为 `(offset, data, seg_index)`，writer 通过 `seg_index` 拿到 `SegmentInfo`。

### ④ checkpoint 保存频率控制

如果每段 fsync 后都调 `_mark_segment_completed` → 每段一次 checkpoint save。对大量小段（如 4MB × 256 = 256 次 save）可能很慢。

**建议**：合并 checkpoint——缓存完成标记，只有收到终止信号或累计 N 段时才写磁盘。或利用 `threading.Timer` 做定期批量保存。

### ⑤ 多级 checkpoint 一致性

- Term 级 checkpoint（ChunkAssembler 内部，per-segment 粒度）
- Segment 级 checkpoint（writer 负责，跨 segment 粒度）

修改只影响 segment 级。term 级独立性不受影响 ✅。但续传时两层 checkpoint 可能不一致：
- Term checkpoint 说段内 80% 完成
- Segment checkpoint 说该段未完成
- → 续传时 term checkpoint 帮助段内跳过已完成的 term，segment checkpoint 触发整段重做

这其实是正确行为：segment 未确认 → 整段重做 → term checkpoint 帮助快速跳过已下载的 term。✅

### ⑥ 模式 ② 的 gap 问题

模式 ② 没有 writer 线程，当前是：
```python
self._process_segment(seg)       # 含 copyfileobj
self._mark_segment_completed(seg)
```

同样有 gap。修复：
```python
with open(self.output_path, 'r+b') as f:
    f.seek(seg.start)
    with open(seg_temp, 'rb') as src:
        shutil.copyfileobj(src, f)
    f.flush()
    os.fsync(f.fileno())        # ★ 保证落盘
# 然后才 mark_completed
```

## 总体结论

| 方面 | 结论 |
|------|------|
| 设计合理性 | 合理。"写入成功才算进度"原则正确 |
| 续传可靠性 | 明显提升。消除了 put/mark 间隙 |
| 进度准确性 | 明显提升。单源上报，无竞态覆盖 |
| 实现风险 | 中等。writer 瓶颈 + 崩溃感知 + checkpoint 频率需处理 |
| 回退代价 | 低。仅在 writer 线程内调整，不改 ChunkAssembler |

**建议先做**：模式 ③ 的 writer 改造（+崩溃传播 + 异步 checkpoint），再对齐模式 ② 的 fsync + mark_completed 顺序。
