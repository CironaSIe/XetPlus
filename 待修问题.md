class CheckpointManager:
    def mark_term_completed(self, file_hash: str, term_idx: int, xorb_hash: str):

        if term_idx % 10 == 0:

### 8. 完成速率估算器（来自 xet.py + Rust）⭐
**优先级**: **P1** - 建议 1 周内完成

**背景**:
- 当前实现：固定水位线预取（48MB-192MB）
- xet.py/Rust 实现：根据完成速率动态调整预取大小
- 收益：低速网络减少预取，高速网络增加预取，内存利用更优

**参考实现**: 
- `~/xet.py/xet/utils.py:ExpWeightedMovingAvg`
- Rust `xet_data/file_reconstruction/reconstruction_terms/manager.rs`

**实现方案**:
```python
# 新建 xet/pipeline/completion_rate_estimator.py
class CompletionRateEstimator:
    """指数加权移动平均速率估算器"""
    
    def __init__(self, half_life: int = 3):
        self.alpha = 0.5 ** (1.0 / half_life)
        self._value = 0.0
        self._last_update = time.time()
    
    def update(self, bytes_delta: int):
        """更新已完成字节数"""
        now = time.time()
        elapsed = now - self._last_update
        
        if elapsed > 0:
            instant_rate = bytes_delta / elapsed
            self._value = self.alpha * self._value + (1 - self.alpha) * instant_rate
        
        self._last_update = now
    
    def get_rate(self) -> float:
        """获取当前速率（bytes/s）"""
        return self._value
```

**自适应预取逻辑**:
```python
# 在 chunk_assembler.py::_prefetch_upcoming_xorbs
completion_rate = self._rate_estimator.get_rate()  # bytes/s

target_buffer_time = 60.0  # 60秒缓冲

# 预取到高水位或目标缓冲区（取较大者）
prefetch_target = max(self.prefetch_high_mb * 1024 * 1024, target_buffer_bytes)
```

### 9. RTT/带宽在线预测（来自 Rust）
**优先级**: **P2** - 建议 1 个月内完成

**背景**:
- 当前实现：ACC 基于成功率调整并发（简单 EWMA）
- Rust 实现：在线线性回归预测 RTT，基于预测调整并发
- 收益：更精确的并发控制，避免过载

**参考实现**: Rust `xet_client/concurrency_controller.rs:OnlineLinearRegression`

**实现方案**:
```python
# xet/network/adaptive_concurrency.py 增强
class AdaptiveConcurrencyController:
    def __init__(self, ...):
        # 现有代码...
        
        # 新增: RTT 预测器
        self._rtt_predictor = OnlineLinearRegression(
            max_samples=100,
            target_max_rtt=5.0,  # 目标最大 RTT (秒)
        )
    
    def report_success(self, rtt: float, bytes_transferred: int):
        """报告成功（新增 RTT 参数）"""
        # 更新 RTT 预测
        self._rtt_predictor.add_sample(
            size=bytes_transferred,
            concurrency=self._current,
            observed_rtt=rtt,
        )
        
        # 预测下次 RTT
        predicted_rtt = self._rtt_predictor.predict(
            size=bytes_transferred,
            concurrency=self._current + 1,
        )
        
        # 基于预测调整并发
        if predicted_rtt < self._target_max_rtt:
            self._adjust_concurrency(delta=+1)
```

### 10. 虚拟许可机制（来自 Rust）
**优先级**: **P2** - 建议 1 个月内完成

**背景**:
- 当前实现：标准信号量（FIFO 等待）
- Rust 实现：虚拟许可（seed permit），下载启动时临时增加许可
- 收益：避免下载启动时的 FIFO 等待，减少启动延迟 10-20%

**参考实现**: Rust `permit::SemaphoreExt::increment_permits_to_target`

**实现方案**:
```python
# 新建 xet/pipeline/virtual_permit.py
class VirtualPermit:
    """可分割的虚拟许可"""
    
    def __init__(self, semaphore: threading.Semaphore, size: int):
        self._semaphore = semaphore
        self._remaining = size
    
    def split(self, size: int) -> Optional['VirtualPermit']:
        """分割许可（不等待）"""
        if size <= self._remaining:
            self._remaining -= size
            return VirtualPermit(self._semaphore, size)
        return None
```

**修改文件**:
- 新建 `xet/pipeline/virtual_permit.py` (+120 行)

**状态**: 已实现并集成

**问题**: 之前的实现一次性下载所有 xorb 到内存（xorb_data_map），大文件会导致内存溢出

**xet.py的设计**:
- 水位线控制：低水位48MB，高水位192MB
- 按需下载和解压，不是一次性全部下载
- 自动释放已使用的 xorb
- 内存占用可控

**实现位置**:
- `xet/pipeline/chunk_assembler.py` - 预取模式核心逻辑
- `xet/pipeline/chunk_assembler_helpers.py` - 预取辅助方法
- `xet/pipeline/file_reconstructor.py` - 使用预取模式
- `xet/cli/commands/download.py` - 添加 `--prefetch-low`, `--prefetch-high` 参数

**功能验证**:
- ✅ 按 term 顺序处理，按需下载 xorb
- ✅ 水位线控制（低水位 48MB，高水位 192MB）
- ✅ 异步预取后续 xorb
- ✅ 自动释放不再需要的 xorb
- ✅ 磁盘缓存集成
- ✅ 添加命令行参数控制

**使用方式**:
```bash
# 默认配置（低水位 48MB，高水位 192MB）
python -m xet.cli.commands.download user/repo/model.gguf

# 低内存环境（降低水位线）
python -m xet.cli.commands.download user/repo/model.gguf \
    --prefetch-low 24 \
    --prefetch-high 96

# 高内存环境（提高水位线）
python -m xet.cli.commands.download user/repo/model.gguf \
    --prefetch-low 96 \
    --prefetch-high 384
```

**设计要点**:
- 按文件顺序处理 terms，不是按 xorb 顺序
- 下载阶段就控制内存，不是只在解压阶段控制
- 压缩数据不保存在内存中，立即解压后释放
- LRU 策略：优先释放后续不再需要的 xorb
- 与磁盘缓存协同：优先从缓存加载

**对比旧实现**:
- 旧：`download_all_xorbs()` → 所有压缩 xorb 保存在 `xorb_data_map` → OOM
- 新：按需下载 → 立即解压 → 使用完释放 → 内存可控
