## 第一部分：数据正确性（Design）

### 核心问题：下载管道没有端到端校验闭环

当前整条管道的校验链条：

```
下载 xorb → 解压 → extract_range → f.write(segment) → per-term SHA256（自校验）
                                                                    ↓ 崩溃/续传
                                                              _verify_resume_terms()
                                                                    ↓ 下载完成
                                                              文件级 SHA256（外部锚点）
```

**最大缺口**：下载解压出的数据是否正确，在写入前**没有任何校验**。唯一能发现"下载就是错数据"的只有最终的文件级 SHA256，但那时已经不知道哪里错了。

**xorb 跨文件共享的现实**（实测发现）：同一个 xorb 的所有 chunk 很可能存在于同一仓库里不同的 XET 文件中。这意味着单文件下载时，一个 xorb 的 chunk 可能只有部分出现在当前文件的 reconstruction 中。xorb_hash 验证需要拿到该 xorb 的**全部** chunk 才能计算，单文件场景下几乎不可行。

### D1 — xorb_hash 端到端验证 + 跨文件补全（最高优先级）

**影响**：①②③
**组件**：`chunk_assembler.py`、`cas_client.py`、`file_reconstructor.py`

**现状**：term 循环（line 891+）中，解压后直接写入，没有调用 `compute_data_hash()` 也没有积累 `(hash, size)` 对。`per-term SHA256` 是自校验（自己算自己存，无法发现"下载就是错数据"的情况）。**唯一有外部锚点的中间验证是 xorb_hash**（`term.hash` 由服务端提供），但需要该 xorb 的所有 chunk。

**已有基础**：
- `xet/storage/merkle_hash.py`：`compute_data_hash()`、`xorb_hash()`、`file_hash()` 已实现并通过测试
- `检查线索.md`：已设计三个注入点（① chunk_hash 积累、② xorb_hash 验证、③ file_hash 验证）
- `~/tmp/opencode/xet_hash_chain.py` + `xet_verify.py`：Python hash 链已验证通过
- `~/tmp/opencode/hash_test/`：Rust 参考实现，可交叉验证
- `CASReconstructionTerm.hash`：服务端提供的 xorb MerkleHash（64 hex 字符），**这是验证锚点**

**核心认识**：
```
验证层级（按粒度从细到粗）：

chunk_hash (compute_data_hash_bytes)  ← 新增，无独立锚点，只用于向上积累
    ↓ 积累
xorb_hash (term.hash)                  ← 新增，有服务端锚点（term.hash），~64MB 粒度
    ↓ 积累
file SHA256 (X-Linked-ETag)            ← 已有，外部锚点，验证整个文件字节流
```

**xorb_hash 的价值**：
1. **早期错误检测**：不用等整个文件下载完才发现数据错了（一个 xorb 完成就能验证）
2. **精确定位**：知道是哪个 xorb（哪些 term）坏了，而不是"整个文件 SHA256 不匹配"
3. **可信的 per-term SHA256**：xorb 验证通过后，生成的 per-term SHA256 是可信的
4. **修复效率**：只需重下损坏的 xorb，而不是整个文件

**跨文件共享的现实**（实测发现）：
同一个 xorb 的所有 chunk 很可能分布在同一仓库里不同的 XET 文件中。例如：
```
xorb_A 有 1000 个 chunk (chunk[0] 到 chunk[999])
  file_A: term[0] 用了 chunk[100:200]   ← 只有 10%
  file_B: term[5] 用了 chunk[500:600]   ← 只有 10%
  file_C: term[3] 用了 chunk[0:100] + chunk[200:500] + chunk[600:1000]  ← 剩余 80%
```

要验证 xorb_A 的 `term.hash`，必须拿到所有 1000 个 chunk。

**策略**：**默认激活 xorb 验证，主动跨文件补全**

代价：可能需要多下载 50%-100% 的数据（取决于 xorb 跨文件共享率）
收益：每个 xorb 验证通过后，该 xorb 的所有 chunk 都是正确的，大幅降低文件级 SHA256 失败的概率

**目标流程**：

```
Phase 0 — 预构建全仓库 xorb 映射表（一次性，可缓存）:
  for each file in repo:
    获取 reconstruction 信息
    提取每个 term 的 (xorb_hash, chunk_range, file_offset)
  
  构建反向索引:
    repo_xorb_map[xorb_hash] = {
      "total_chunks": 100,
      "files": {
        "file_A": [{"term_idx": 0, "chunk_range": (0, 10), "file_offset": ...}],
        "file_B": [...],
        ...
      }
    }
  
  缓存到 .xet/cache/repo_xorb_map_{commit_sha}.json

Phase 1 — 下载当前文件 + xorb 补全:
  for each term:
    解压 chunk → compute_data_hash(uncompressed) → 积累到 xorb_accumulator[term.hash]
    write segment → file
    
    检查 xorb 是否完整:
      if len(xorb_accumulator[term.hash]) < repo_xorb_map[term.hash]["total_chunks"]:
        # 查表：缺失的 chunk 在哪些文件
        for other_file, other_term in repo_xorb_map[term.hash]["files"]:
          下载 other_file 的 other_term 的 xorb 数据
          解压缺失的 chunk → 积累到 xorb_accumulator[term.hash]
      
      # 现在 xorb 完整了
      computed_xorb_hash = xorb_hash_from_hashes(xorb_accumulator[term.hash])
      if computed_xorb_hash != term.hash:
        raise ReconstructionError(f"xorb {term.hash[:16]} 验证失败")
      
      标记该 xorb 为"已验证"
      为当前 term 生成可信的 per-term SHA256

Phase 2 — 文件级 SHA256 验证:
  所有 term 处理完后:
    computed_sha256 = hashlib.sha256(file).hexdigest()
    if computed_sha256 != expected_sha256:
      # 大概率不会走到这里（所有 xorb 都验证过了）
      报错 + 进入修复流程
```

**内存开销**：
- repo_xorb_map：假设 20000 个 xorb，每个 ~1.3 KB → ~26 MB（可接受）
- xorb_accumulator：每 chunk 68 bytes (64B hex + 4B size)。4K chunks ≈ 272 KB（完全流式）

**进度条需求**：
1. **构建 repo_xorb_map**（一次性，~100 秒 for 1000 文件）：
   ```
   正在构建仓库 xorb 映射表...
   ████████░░░░░░░░░░░░ 42% (420/1000 文件)
   ```

2. **下载 + 补全**：
   ```
   下载 large_model.bin...
   ████████████████░░░░ 80% (80/100 term)
   补全 xorb_A... ████████████████████ 100% (50 MB / 50 MB)
   ```

**方案要点**：
1. **Phase 0**：预构建 `repo_xorb_map`（可选，但强烈推荐）——如果不做，只能在单文件内验证完整的 xorb
2. **Phase 1**：在 term 循环中注入 chunk_hash 积累 + xorb 补全逻辑
3. 需要按 chunk 粒度解压——新增 `StreamingXorbAccessor.extract_chunks_in_range()` 方法
4. 分段模式下 xorb_accumulator 需跨段共享（由 `SegmentedReconstructor` 持有）
5. 每个 xorb 验证通过后，立即标记为"已验证"，避免重复验证

### D2 — 并行写入偏移为累加值，无校验

**影响**：③
**组件**：`chunk_assembler.py:1103-1195` `_assemble_with_parallel_write`

**现状**：`current_offset += len(segment)`（line 1195）盲信式累加。如果任一 term 的 `extract_range` 返回长度与预期不符（比如 xorb 下载不完整导致 `end_byte` 查找到错误边界），后续所有偏移全错。

**对比**：顺序模式用 `f.tell()`（line 969），是真实文件指针位置，天然正确。

**更严重的问题**：并行模式的续传**没有调用 `_verify_resume_terms()`**（顺序模式 line 727-751 有调用）。如果崩溃导致 .part 文件尾部损坏，并行模式不会检测到。

**方案**：
1. 并行模式续传恢复加入 `_verify_resume_terms()` 调用
2. 写入完成后用总字节数校验 `current_offset` 是否等于 `sum(term.unpacked_length) - offset_into_first_range`，不等则报错
3. 或考虑：并行模式的偏移计算也改用类似顺序模式的"第一个 term 用 writer 线程的 `f.tell()`，后续累加但有最终校验"

### D3 — 修复逻辑需基于 xorb 验证 + 精确覆写

**影响**：①②③
**组件**：`file_verifier.py:357-437` `_repair_xorb`

**现状**：

```python
# file_verifier.py:435-436
f.seek(record.file_offset)  # ← 用 checkpoint 中存的偏移
f.write(segment)
```

修复逻辑依赖 per-term SHA256 诊断损坏位置，但 per-term SHA256 是自校验（无外部锚点），无法检测"下载时就是错数据"的情况。

**另一个问题**：修复后没有验证写入数据的正确性——重新下载的 xorb 也可能是错误数据。

**新流程（基于 xorb 验证）**：

```
文件级 SHA256 失败
  ↓
逐个 xorb 验证（用 repo_xorb_map 补全缺失的 chunk）:
  for xorb_hash in current_file_xorbs:
    if xorb_hash not in verified_xorbs:
      补全该 xorb 的所有 chunk（从全仓库下载）
      computed = xorb_hash_from_hashes(all_chunks)
      if computed != xorb_hash:
        corrupted_xorbs.add(xorb_hash)
  ↓
for xorb_hash in corrupted_xorbs:
  # 查表：该 xorb 的完整数据在哪些文件
  locations = repo_xorb_map[xorb_hash]["files"]
  
  # 重新下载该 xorb 的所有数据（跨文件）
  all_chunks = {}
  for file_path, term_list in locations.items():
    for term_info in term_list:
      下载并解压该 term 的 xorb 数据
      积累到 all_chunks
  
  # 验证重新下载的 xorb
  computed = xorb_hash_from_hashes(all_chunks.values())
  if computed != xorb_hash:
    raise ReconstructionError(f"重新下载的 xorb {xorb_hash[:16]} 仍然损坏，放弃修复")
  
  # 精确覆写：只覆写当前文件中用到该 xorb 的字节范围
  for term in current_file.terms:
    if term.hash == xorb_hash:
      file_offset = calculate_term_offset(term)
      segment = extract_range_from_chunks(all_chunks, term.chunks)
      f.seek(file_offset)
      f.write(segment)
      logger.info(f"已修复 term #{term_idx}, offset={file_offset}, len={len(segment)}")
  ↓
重新验证文件级 SHA256
  ↓
如果仍然失败 → 放弃修复，建议全量重下
```

**关键点**：
1. **不截断文件**：只覆写损坏的字节范围，保留其他正确的数据
2. **依赖 xorb 验证**：不依赖 per-term SHA256（它没有外部锚点）
3. **跨文件补全**：用 `repo_xorb_map` 获取 xorb 的完整数据
4. **验证重新下载的数据**：确保修复后的 xorb 是正确的
5. **精确定位**：知道哪个 xorb 坏了 → 知道文件中哪些字节范围需要重写

**进度条**：
```
修复 3 个损坏的 xorb...
████████░░░░░░░░░░░░ 33% (1/3 xorb)
重新下载 xorb_C (50 MB)... ████████████████████ 100%
覆写 file_A: term[5] @ offset 3200000...
```

### D4 — per-term SHA256 角色需重新定位

**影响**：①②③
**组件**：`chunk_assembler.py`、`file_verifier.py`、`checkpoint_manager.py`

**现状**：per-term SHA256 承担两件事：
1. 续传回溯定位（`_verify_resume_terms()`）：**有价值**
2. 写入后损坏诊断（`file_verifier._run_diagnosis()`）：**价值有限**

诊断的局限性（`file_verifier.py:228-232` 已承认）：
> "文件级 SHA256 不匹配但 per-term 诊断无损坏项。这可能意味着下载时就收到了错误数据。"

**未来定位**：
- 如果 D1（file_hash 端到端校验）到位，per-term 诊断的价值大幅降低
- file_hash 不匹配时，可以直接按 xorb 分组的 chunk_hash 定位问题 chunk（chunk_hash 有服务端锚点 via Merkle tree）
- per-term SHA256 保留用于续传，但降级为可选
- checkpoint 可以简化：不再需要存每个 term 的 SHA256（续传用 `confirmed_bytes` + 文件大小估算就够了）

**不急于改**：在 D1 实现后再决定是否简化 checkpoint 格式。

**影响**：①②③
**组件**：`chunk_assembler.py:930-947`、`chunk_assembler.py:1144-1164`、`file_verifier.py:409-420`

**现状**：当 `end_chunk` 不在 `chunk_offsets_dict` 中时（xorb 下载不完整），需要遍历整个 dict 找下一个：

```python
if end_chunk not in chunk_offsets_dict:
    sorted_chunks = sorted(chunk_offsets_dict.items())
    for chunk_id, offset in sorted_chunks:
```

## 第三部分：未来校验流程设计

### 目标状态

```
下载管道校验链（目标）:

  chunk 解压 → compute_data_hash(data) → 积累 (hash, size)
  xorb 完整? → compute_xorb_hash(collected) vs term.hash  (可选，单文件时跳过)
  文件完成   → compute_file_hash(all)   vs entry file_hash (必须)
  文件完成   → compute_sha256(file)      vs expected_sha256  (可选，如有锚点)
```

### 分阶段实施路线

**Phase 1 — file_hash 端到端校验**（最高优先级）：
- 改造 term 循环：每 chunk 解压后调用 `compute_data_hash()`，积累 `(hash_bytes, size)` 对
- 文件组装完成后计算 `file_hash` 对比入口值
- 需要改造 `StreamingXorbAccessor.extract_range()` 使其支持按 chunk 粒度返回解压数据
- 内存开销：~272KB/4K chunks，可接受

**Phase 2 — 修复流程改造**：
- `_repair_xorb()` 改为截断+重建模式
- 修复后必须验证 file_hash
- file_hash 仍不匹配时放弃修复，建议全量重下

**Phase 3 — per-term SHA256 角色调整**（Phase 1+2 完成后评估）：
- 评估是否可以简化 checkpoint 格式
- 如果 file_hash 校验足够可靠，per-term SHA256 可能只用于续传

### xorb_hash 验证的定位

由于 xorb 跨文件共享（同一 xorb 的 chunk 分布在同一仓库不同 XET 文件中），单文件下载时 xorb_hash 验证几乎不可行。

**2026-06-29 确认**：BATCH API 只返回当前请求文件引用的 chunk，无法通过公开 API 获取完整 xorb。跨文件补全不可行。

xorb_hash 验证只在数据巧合完整时做（如完整 xorb `142d6e57`，906 个 chunk 全部在当前文件中）。

### data_hash 逐 chunk 校验

**2026-06-29 确认**：xorb transfer URL 只返回 chunk data（不包含 boundaries footer），无法获取期望的 per-chunk `chunk_hashes` 列表。Python 层无法做逐 chunk 的 `data_hash` 比对。

可行的间接校验：Merkle 聚合所有 chunk 的 `data_hash` → `xorb_hash` → 比对。已验证通过（906/906 chunk）。

## D1 渐进式校验缺失

### Rust 有但 Python 没有的校验能力

Rust 的 `MerkleHashSubtree`（`xet_core_structures/src/merklehash/merkle_hash_subtree.rs`）支持渐进式 chunk hash 验证：

- **逐 chunk data_hash 校验**：Rust 全量反序列化时检查每个 chunk 的 `compute_data_hash(data) == chunk_hashes[idx]`（`xorb_object_format.rs:1112-1119`），Python `StreamingXorbAccessor._decompress_single_entry` 不做此检查
- **MerkleHashSubtree**：O(log n) 空间存储部分聚合的 Merkle 树"驼峰"，支持 `from_chunks`→`merge_into`→`final_hash` 渐进式验证
- **Python `merkle_hash.py`** 只有 `aggregated_node_hash`（O(n) 全量聚合），没有渐进式合并能力

### 影响

当前 D1 实现只能在所有 term 下载完后一次性算 `file_hash`，不能在下载过程中逐步验证已到达 chunk 的数据完整性。这导致：
1. 内存 O(n) 累积全部 chunk 数据/hash
2. 无法提前发现 corruption（必须等全部下载完才知道）
3. 未复用的 `_complete_xorb_cross_file` 做跨文件补全（API 限制，不可行）

### 建议修复方向

如有需要，从 Rust `merkle_hash_subtree.rs` 移植 `MerkleHashSubtree` 到 Python，替换当前的 `xorb_accumulator` 全量列表收集模式。优先级低 — 当前 `file_hash` 端到端校验已能覆盖数据完整性。
