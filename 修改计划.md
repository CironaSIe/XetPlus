# 修改计划

## 目标

解决三类核心问题：

1. **CAS URL 刷新逻辑错误** → 消除周期性 403 和"跑 30s 卡 30s"
2. **续传丢分片** → 消除 put/mark 间隙，确保 checkpoint 与落盘一致
3. **并行进度错乱** → writer 单源上报，消除竞态

---

## 阶段一：CAS 刷新重写（`cas_client.py` + `url_refresh_coordinator.py`）

### 1.1 移除 pre-refresh（CAS 预检刷新）

**位置**：`cas_client.py:600-618`

**现状**：`get_xorb_data_with_retry()` 每 30s 在 attempt 0 时调用 `get_reconstruction()` 预刷新。但：
- 走 `_recon_cache` 拿相同 URL（缓存 TTL = 30s，每次预刷新触发时缓存刚好过期 → 调 API，但 403 handler 也调，重复浪费）
- 异常被 `except Exception: pass` 吞掉 → `current_url` 留旧值 → 403
- 共享状态 `_last_recon_time`、`_last_recon_file` 无锁保护

**改法**：直接删除 pre-refresh 代码块（line 600-618），URL 刷新只保留在 403 handler 中。

```diff
- # 2.5. 预检 URL 新鲜度...
- _need_fresh_recon = (
-     file_hash != self._last_recon_file
-     or time.time() - self._last_recon_time > CASClient._recon_refresh_interval
- )
- if _need_fresh_recon and attempt == 0:
-     try:
-         _fresh_recon = self.get_reconstruction(file_hash)
-         _fresh_url, _fresh_range = self._find_xorb_in_recon(...)
-         current_url = _fresh_url
-         current_range = _fresh_range
-         self._last_recon_time = time.time()
-         self._last_recon_file = file_hash
-     except Exception:
-         pass
```

**连带删除**：`_last_recon_time`、`_last_recon_file` 属性及使用它们的全部代码。

### 1.2 403 handler 优化

**位置**：`cas_client.py:673-740`

**现状**：
- 调 `_force_refresh_token()`：每次 403 都刷 token（可能不必要）
- 清 `_recon_cache` 后调 `get_reconstruction()`
- 写入 `_shared_refresh_recon`
- 专用退避 `5.0 * (2.5^attempt)`，最长 ~62s

**改法**：

a) **区分"真过期"和"假过期"**：添加 `_last_403_time` 跟踪。连续两次 403 间隔 < 30s，且未改 URL → 判定为限流 → 延长退避 + 通知 ACC 收缩并发。

b) **减少 token 刷新**：403 handler 先尝试清缓存 + 调 `get_reconstruction()` 获取新 URL。如果还 403，再考虑 token 是否过期。

c) **统一共享路径**：pre-refresh 移除后，`_shared_refresh_recon` 只在 403 handler 中写入，清晰且唯一。

### 1.3 pre-refresh 移除后的影响

| 影响 | 评估 |
|------|------|
| 第一个 xorb 的 URL | 来自 attempt 0 的 `url` 参数（调用者传入的原始 URL）。如果已过期 → attempt 0 403 → 403 handler 刷新。 |
| 串行模式(①) | 无 pre-refresh 后不再有"跑 30s 卡 30s" | 
| 并行模式(②③) | 同串行，不再浪费 API 调用 |
| 初始 403 率 | 略有增加（第一个 xorb 的原始 URL 可能刚过期）。但只在中断后恢复下载时发生，正常连续下载不受影响。 |

**可选的缓和措施**：在 `get_reconstruction()` 调用完成后，主动更新 `_recon_cache` 为最新数据，确保后续 xorb 的 attempt 0 使用新 URL（通过 `_shared_refresh_recon` 检查）。即使 pre-refresh 移除，`_shared_refresh_recon` 仍然在 attempt 0 入口处检查（line 556-570）：

```python
with self._shared_refresh_lock:
    if self._shared_refresh_recon is not None:
        ...
        current_url, current_range = self._find_xorb_in_recon(...)
```

这样第一个遇到 403 的线程刷新的 URL 会自动共享给其他线程。

### 1.4 _shared_refresh_recon 增强

**现状**：只在 403 handler 中写入，TTL 120s。

**改法**：在 `get_xorb_data_with_retry` 成功下载后（line 635 `return data` 前），将当前使用的 reconstruction 写入 `_shared_refresh_recon`。这样后续 xorb 的 attempt 0 可以直接使用最新 URL，避免 403。

## 阶段二：Writer 线程改造（`segmented_reconstructor.py` + `global_writer.py`）

### 2.1 Writer 承担 mark_completed 职责

**原则**：写入成功（fsync）后，writer 线程负责标记段完成。

**队列元组改为**：`(offset: int, data: bytes, seg_index: int)`

**writer 流程**：

```python
def _writer_worker(self):
    with open(self.output_path, 'r+b') as f:
        while True:
            item = self._write_queue.get()
            if item is None:
                break
            offset, data, seg_index = item
            f.seek(offset)
            f.write(data)
            f.flush()
            os.fsync(f.fileno())
            self._total_written += len(data)
            # Writer 负责标记完成
            self._mark_segment_completed_by_index(seg_index)
            # Writer 是 bytes 和段进度的唯一来源
            self._report_progress()
```

**新增方法** `_mark_segment_completed_by_index(seg_index)`：
```python
def _mark_segment_completed_by_index(self, seg_index: int):
    with self._lock:
        seg = self.segments[seg_index]
        if seg.completed:
            return  # 已标记过，防重复
        seg.completed = True
        self.completed_segments.append({
            'index': seg.index, 'start': seg.start,
            'end': seg.end, 'size': seg.size,
        })
        self.completed_set.add((seg.start, seg.end))
        # 异步保存 checkpoint
        self._schedule_checkpoint_save()
```

### 2.2 Worker 移除进度上报

`_process_segment_parallel` 只做：download → decompress → put，**不再调用 progress_callback**。

```python
def _process_segment_parallel(self, seg):
    recon = self._load_recon_cache(seg) or self.cas_client.get_segment_reconstruction(...)
    segment_data = self._download_and_assemble_segment(seg, recon)
    self._write_queue.put((seg.start, segment_data, seg.index))
    # 不再调 self.progress_callback
    # 不再调 self._mark_segment_completed(seg)
```

`_download_and_assemble_segment` 内部的 ChunkAssembler ProgressTracker 保持（用于段内进度），但其回调直接丢弃或只用于日志：

```python
# 并行模式下，ChunkAssembler 的 ProgressTracker 回调改为 no-op
_worker_cb = lambda stats: None
```

### 2.3 异步 checkpoint 保存

**目的**：防止 checkpoint I/O 阻塞 writer 写入循环。

**方案一：Timer 批量保存**
```python
def __init__(self, ...):
    self._checkpoint_dirty = False
    self._checkpoint_timer: Optional[threading.Timer] = None

def _schedule_checkpoint_save(self):
    self._checkpoint_dirty = True
    if self._checkpoint_timer is None:
        self._checkpoint_timer = threading.Timer(1.0, self._flush_checkpoint)
        self._checkpoint_timer.daemon = True
        self._checkpoint_timer.start()

def _flush_checkpoint(self):
    with self._lock:
        if self._checkpoint_dirty:
            self.segment_checkpoint.save(
                file_hash=self.file_hash,
                completed_segments=self.completed_segments,
            )
            self._checkpoint_dirty = False
        self._checkpoint_timer = None
```

**方案二（更简单）：写在 writer 外**
writer 写完一批数据后，不直接调 save，而是在 `_writer_worker` 中每 N 次迭代或收到终止信号时，在 `f.close()` 之后才 save checkpoint。

```python
def _writer_worker(self):
    write_count = 0
    with open(...) as f:
        while True:
            item = self._write_queue.get()
            if item is None:
                break
            ...
            write_count += 1
    # 文件已关闭，安全保存 checkpoint
    self._save_final_checkpoint()
```

但这种方式不能保证 fsync 后实时保存。适合 checkpoint 频率低、容忍丢失部分的场景。

### 2.4 Writer 崩溃传播

**现状**：daemon 线程异常不传播。

**改法**：`threading.Event` + 异常变量

```python
class SegmentedReconstructor:
    def __init__(self, ...):
        ...
        self._writer_exception: Optional[Exception] = None
        self._writer_exception_event = threading.Event()
```

```python
def _writer_worker(self):
    try:
        ...
    except Exception as e:
        self._writer_exception = e
        self._writer_exception_event.set()
        logger.error(...)
```

```python
def _reconstruct_parallel(self):
    ...
    with ThreadPoolExecutor(...) as executor:
        ...
        for future in as_completed(future_to_seg):
            # 每次检查 writer 是否崩溃
            if self._writer_exception_event.is_set():
                raise RuntimeError(f"Writer 线程崩溃") from self._writer_exception
            ...
    self._write_queue.put(None)
    self._writer_thread.join()
```

### 2.5 Worker 段下载异常处理

当某个 worker 的段下载失败时：
- 当前行为：`future.result()` 抛出异常，加入 `failed_segments` 列表
- 但 writer 可能已经写入了其他 worker 的段数据
- 其他 worker 还在继续工作

**改法**：任一段失败 → 立刻 stop_event.set() → 通知 writer 和其他 worker 停止。

```python
for future in as_completed(future_to_seg):
    if self._stop_event.is_set():
        for f in future_to_seg:
            f.cancel()
        break
    try:
        future.result()
    except Exception as e:
        self._stop_event.set()  # 通知所有 worker + writer 停止
        failed_segments.append((seg, e))
```

writer 线程也需要检查 `_stop_event`：
```python
def _writer_worker(self):
    while True:
        if self._stop_event.is_set():
            break
        item = self._write_queue.get(timeout=1.0)  # 或非阻塞
        ...
```

### 2.6 模式 ② 对齐

模式 ②（`_reconstruct_sequential`）无 writer 线程，但同样原则适用：

```python
def _process_segment(self, seg):
    ...
    with open(self.output_path, 'r+b') as f:
        f.seek(seg.start)
        with open(seg_temp, 'rb') as src:
            shutil.copyfileobj(src, f)
        f.flush()
        os.fsync(f.fileno())    # ★ 保证落盘
    self._mark_segment_completed(seg)  # ★ 落盘后才标记
```

## 阶段三：进度显示改造（`segmented_reconstructor.py` + `download.py` + `progress_tracker.py`）

### 3.1 进度上报统一架构

| 谁上报 | 上报内容 | 口径 |
|--------|----------|------|
| Writer 线程 | `assembled_bytes`、`completed_segments`、`total_segments` | 文件级 |
| （无） | xorb/term 进度 | 并行模式下不报（因为独立 Tracker 竞态无法解决） |

简化：并行模式下不再显示 xorb/term 级进度，只显示：
- `📦 x MB / y MB (z%)`（bytes，writer 上报）
- `🔗 a / b 段`（segments，writer 上报）
- `⚡ n MB/s`（速率，download.py 计算）
- `⏱ ETA`（剩余时间）

串行模式（① ②）维持现状。

### 3.2 _latest_stats 简化

```python
def progress_callback(stats):
    with _progress_lock:
        # 不再 update，只写入 writer 报的字段
        for k in ('assembled_bytes', 'total_bytes', 'downloaded_bytes',
                    'progress_pct', 'completed_segments', 'total_segments',
                    'speed_bps', 'eta_seconds', 'elapsed_seconds'):
            if k in stats:
                _latest_stats[k] = stats[k]
        # 出锁后更新显示
    progress.update(_latest_stats)
```

### 3.3 断点续传初始进度

```python
if resume and self.progress_callback:

    self.progress_callback({
```

不再检查 `.part` 文件（模式 ③ 没有 .part，数据在 writer 队列中；模式 ② 有 .part 但续传从 segment checkpoint 恢复，更准确）。

## 阶段四：续传可靠性加固

### 4.1 模式 ③ 续传恢复

**现状**：`_reconstruct_parallel` 中 writer 线程写完数据后，`_mark_segment_completed` 异步执行。

**改后**：writer 线程负责 mark + save checkpoint。checkpoint 保存异步化（见 2.3）。

恢复流程：
1. 加载 segment checkpoint → `completed_set`
2. 跳过 `completed_set` 中的段
3. 未完成的段：重建 output 文件（预分配空间）→ ThreadPoolExecutor 并行处理
4. Writer 线程写入后 automatic mark

**不需要** 检查 `.part` 文件。模式 ③ 的输出只有最终的 output 文件。

### 4.2 模式 ② 续传恢复

恢复流程（同现有，但增加 fsync）：
1. 加载 segment checkpoint → `completed_set`
2. 跳过已完成段
3. 未完成段：ChunkAssembler 从 term-level checkpoint 恢复（如果有）
4. 写 `.tmp` → copyfileobj→fsync→mark_completed

### 4.3 预分配文件与大小校验

并行写入模式预分配 output 文件（`f.seek(file_size-1)`）。续传时检查文件大小是否匹配：

```python
if self.parallel_write and self.output_path.exists():
    actual = self.output_path.stat().st_size
    expected = self.file_size
    if actual != expected:
        if actual < expected:
            # 文件不够大，重新预分配
            with open(self.output_path, 'r+b') as f:
                f.truncate(expected)
        # 如果实际 > 预期：不可能（预分配是精确的），但以防万一
```

## 修改顺序

```
阶段一 → 阶段二 → 阶段三 → 阶段四
(CAS)  → (Writer) → (进度) → (续传)
```

**阶段一**：`cas_client.py` + `url_refresh_coordinator.py`
**阶段二**：`segmented_reconstructor.py`（writer + worker + checkpoint + crash）+ `global_writer.py`
**阶段三**：`segmented_reconstructor.py` + `download.py` + `progress_tracker.py`
**阶段四**：在阶段二基础上加固续传路径
