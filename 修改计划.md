## 6. 直写续传 seek 修正

### 当前代码

`chunk_assembler.py:848-851`：

```python
if output_offset > 0:
    # 直写模式：跳到目标偏移
    f.seek(output_offset)
```

不管 `start_term_idx` 是否 > 0，总是 seek 到 `output_offset`（= `seg.start`，段起始偏移）。但上方 line 699-702 **已经从 checkpoint 加载了 `confirmed_bytes`**（= `f.tell()` 保存的绝对文件偏移）。

续传场景：段内已写完 term 0-9（confirmed_bytes = seg.start + 80MB），checkpoint 记录从 term 10 继续。此时 `f.seek(output_offset)` 定位到 seg.start，term 10 的数据会覆盖已完成的 term 0-9。

### 修改方案

将 seek 逻辑改为：`start_term_idx > 0` 时优先使用 `confirmed_bytes`，回退到 terms 累加：

```python
if output_offset > 0:
    seek_pos = output_offset
    if start_term_idx > 0:
        if confirmed_bytes > 0:
            seek_pos = confirmed_bytes
        else:
            for i in range(start_term_idx):
                if i == 0:
                    seek_pos += max(0, recon.terms[i].unpacked_length - recon.offset_into_first_range)
                else:
                    seek_pos += recon.terms[i].unpacked_length
    f.seek(seek_pos)
    logger.debug(f"[ChunkAssembler] 直写模式: offset={output_offset}, seek_pos={seek_pos}")
```

**同时更新 line 759 注释**：删掉"不支持 term 级别断点恢复"，改为"续传时 seek 到 confirmed_bytes"。

### 涉及行

| 文件 | 行 | 变更 |
|------|-----|------|
| `chunk_assembler.py` | 848-851 | 替换为上方逻辑 |
| `chunk_assembler.py` | 759 | 更新注释 |

### 边界

- `output_offset == 0`（普通模式）：不受影响，走 line 852-874 的 r+b 逻辑
- `start_term_idx == 0`：seek_pos == output_offset，行为不变
- `confirmed_bytes == 0 && start_term_idx > 0`：回退到 terms 累加计算

## 7. queue.put 加超时

### 当前代码

`segmented_reconstructor.py:691`：

```python
self._write_queue.put((seg.start, seg.index, seg_temp, seg_size))
```

无超时。如果 writer 崩溃且队列满了，worker 永久阻塞。

**风险**：当前 `maxsize = parallel_segments * 2`，worker 数 ≤ parallel_segments，每个 worker 只 put 1 次 → 正常情况队列不会满。但 writer 崩溃后队列可能接近满（writer 消费了部分但未清空），下次 put 有理论可能阻塞。

### 修改方案

加 `timeout=5.0`，超时后检查 `_stop_event`：

```python
# line 691 替换为：
while True:
    try:
        self._write_queue.put((seg.start, seg.index, seg_temp, seg_size), timeout=5.0)
        break
    except queue.Full:
        if self._stop_event.is_set():
            logger.debug(f"[SegmentedReconstructor] 段 {seg.index} 写入取消（停止信号），丢弃临时文件")
            seg_temp.unlink(missing_ok=True)
            return
        continue
```

需要确认文件顶部已导入 `queue`（已有 `_write_queue = queue.Queue(...)` → 已导入）。

### 涉及行

| 文件 | 行 | 变更 |
|------|-----|------|
| `segmented_reconstructor.py` | 689-691 | 替换为 timeout 循环 |

## 8. _shared_refresh_recon 过期清理

### 当前代码

`cas_client.py:663-666`（403 handler 中写入）：

```python
with self._shared_refresh_lock:
    self._shared_refresh_recon = (file_hash, recon, time.time())
```

写入时不检查旧条目是否过期，`_shared_refresh_recon` 永远不会被置 None。

读取处（line 541-555, 690-705）只做 TTL 判断来选择是否使用，但**过期条目始终驻留在内存**（一个 `QueryReconstructionResponse` 可能几 MB）。

### 修改方案

在写入新值时清理过期旧值：

```python
# line 663-666 替换为：
with self._shared_refresh_lock:
    if self._shared_refresh_recon is not None:
        _, _, old_ts = self._shared_refresh_recon
        if time.time() - old_ts > 120:
            self._shared_refresh_recon = None
    self._shared_refresh_recon = (file_hash, recon, time.time())
```

同时在 `_set_recon_cache`（line 102-109）中顺带清理共享结果：

```python
def _set_recon_cache(self, file_hash: str, recon: QueryReconstructionResponse):
    with self._recon_cache_lock:
        self._recon_cache[file_hash] = (recon, time.time())
        for k in list(self._recon_cache.keys()):
            if k != file_hash:
                del self._recon_cache[k]
    # 顺带清理过期的共享刷新结果
    with self._shared_refresh_lock:
        if self._shared_refresh_recon is not None:
            s_hash, _, s_ts = self._shared_refresh_recon
            if s_hash != file_hash or time.time() - s_ts > 120:
                self._shared_refresh_recon = None
```

| 文件 | 行 | 变更 |
|------|-----|------|
| `cas_client.py` | 663-666 | 写入前清理过期条目 |

## 9. 低速续传不消耗 attempt

### 当前代码

`cas_client.py:537` + `719-741`：

```python
for attempt in range(self.retry_max):
    ...
    except LowSpeedTimeoutError as e:
        ...
        # 低速超时不算入重试次数（只调整范围）
        continue    # ← Python for 循环：attempt 仍递增
```

**关键问题**：Python `for ... in range()` 循环中修改 `attempt` 无效——下次迭代从 range 迭代器取下一个值。`attempt -= 1` 不起作用。

### 修改方案

改为 `while` 循环，手动管理 `attempt`：

```python
attempt = 0
low_speed_count = 0
while attempt < self.retry_max:
    try:
        ...  # 原有 try 块内容
        break   # 成功 → 退出
    except 401:
        ...
        attempt += 1
        if attempt >= self.retry_max:
            raise
        ...backoff...
        continue
    except 403:
        ...
        attempt += 1
        if attempt >= self.retry_max:
            raise
        ...backoff...
        continue
    except LowSpeedTimeoutError as e:
        ...  # 原有低速续传逻辑（不递增 attempt）
        low_speed_count += 1
        if low_speed_count > self.retry_max * 5:  # 安全阀
            raise RuntimeError(f"低速续传累计 {low_speed_count} 次，放弃")
        continue
    except requests.RequestException as e:
        ...
        attempt += 1
        if attempt >= self.retry_max:
            raise
        ...backoff...
        continue
```

**迁移要点**：
- 所有 `if attempt == self.retry_max - 1:` 改为 `if attempt + 1 >= self.retry_max:`
- 所有 `attempt + 1` 日志中的 display 值保持 `attempt + 1`
- 退避公式 `1.5 ** attempt` / `2.5 ** attempt` 仍用 `attempt`（仅计真实重试次数，更合理）
- 添加 `low_speed_count` 安全阀避免无限低速续传

### 涉及行

| 文件 | 行 | 变更 |
|------|-----|------|
| `cas_client.py` | 537 | `for attempt in range(self.retry_max):` → `while attempt < self.retry_max:` |
| `cas_client.py` | 多处 | 各 excep handler 补 `attempt += 1`，条件判断调整 |
| `cas_client.py` | 719-741 | LowSpeedTimeoutError 不递增 attempt，加安全阀 |

## 11. _save_recon_cache 并发保护

### 当前代码

`segmented_reconstructor.py:817-835` 和 `783-815`：

- `_save_recon_cache`：`open(cache_path, 'w')` 无锁
- `_load_recon_cache`：`open(cache_path, 'r')` 无锁

正常流程：不同段写不同 `seg_N.recon.json` → 无冲突。仅在 future 取消+重试同段时可能并发读写同一文件。

```python
def _load_recon_cache(self, seg: SegmentInfo):

    if not cache_path.exists():
        return None
    with self._lock:
        try:
            with open(cache_path, 'r', encoding='utf-8') as f:
                data = json.load(f)
            ...  # 原有验证逻辑
        except ...:
            return None

def _save_recon_cache(self, seg: SegmentInfo, recon) -> None:
    cache_path = self._get_recon_cache_path(seg)
    with self._lock:
        try:
            ...  # 原有写入逻辑
        except Exception as e:
            logger.warning(...)
```

**注意**：`self._lock` 目前被 writer 和 segment 进度共享（`_mark_segment_completed`、`_mark_segment_completed_by_index` 等）。recon cache 操作很快（~1ms），不会增加锁竞争。如果担心，可改用专用 `_recon_cache_lock = threading.Lock()`。

## 12. _save_unsafe 原子写入

### 当前代码

`checkpoint_manager.py:289-308`：

```python
def _save_unsafe(self, checkpoint):
    ...
    with open(self.checkpoint_path, 'w', encoding='utf-8') as f:
        json.dump(data, f, indent=2)
    self._cache = checkpoint
```

直接写目标文件。崩溃窗口内（kill -9 / 掉电），checkpoint 文件可能截断/损坏 → 下次 `json.load()` 抛异常 → 整个 checkpoint 丢失。

对照：`SegmentCheckpointManager.save()`（`segmented_reconstructor.py:117-149`）**已经使用了 tmp+replace 模式**——先写临时文件再原子 replace。

### 修改方案

复用 `SegmentCheckpointManager` 的原子写入模式：

```python
def _save_unsafe(self, checkpoint: ReconstructionCheckpoint) -> None:
    if not self.checkpoint_path:
        return
    try:
        self.checkpoint_path.parent.mkdir(parents=True, exist_ok=True)
        data = checkpoint.to_dict()
        tmp_path = self.checkpoint_path.with_suffix('.tmp')
        with open(tmp_path, 'w', encoding='utf-8') as f:
            json.dump(data, f, indent=2)
            f.flush()
            os.fsync(f.fileno())
        tmp_path.replace(self.checkpoint_path)
        self._cache = checkpoint
    except Exception as e:
        logger.error(f"[Checkpoint] 保存失败: {e}")
```

需确认 `checkpoint_manager.py` 顶部已导入 `os`（当前未导入 → 需添加）。

### 涉及行

| 文件 | 行 | 变更 |
|------|-----|------|
| `checkpoint_manager.py` | 5 | 添加 `import os` |
| `checkpoint_manager.py` | 289-308 | 替换为 tmp+replace |

## 13. ACC 信号量漂移修正

### 当前代码

`adaptive_concurrency.py`：

- `_maybe_increase`（line 199-200）：`self._semaphore.release()` 增加一个许可
- `_maybe_decrease`（line 226-228）：只 `self._current -= 1`，不 `acquire`
- `release()`（line 112-114）：直接 `self._semaphore.release()`

increase 比 decrease 多做了一次 `semaphore.release()` → 信号量实际许可数 > `_current` → 长期累积后失去限流效果。

### 修改方案

加 `_pending_acquire` 标记，让下次 `release()` 时消费这个"欠债"：

**`__init__`（line 77-79 之间）**添加：

```python
self._pending_acquire: bool = False  # _maybe_decrease 欠债标记
```

**`_maybe_decrease`（line 226 之后）**添加：

```python
self._pending_acquire = True
```

**`release`（line 112-114）**修改为：

```python
def release(self):
    with self._lock:
        if self._pending_acquire:
            self._pending_acquire = False
        else:
            self._semaphore.release()
```

**`reset`（line 248-264）**也需要重置标记：

```python
def reset(self):
    with self._lock:
        ...
        self._pending_acquire = False
```

### 涉及行

| 文件 | 行 | 变更 |
|------|-----|------|
| `adaptive_concurrency.py` | ~79 | `__init__` 增加 `_pending_acquire` |
| `adaptive_concurrency.py` | 226 | `_maybe_decrease` 增加 `self._pending_acquire = True` |
| `adaptive_concurrency.py` | 112-114 | `release` 改为有条件释放 |
| `adaptive_concurrency.py` | 248-264 | `reset` 中重置 `_pending_acquire` |

## 14. truncate 前断言保护

### 当前代码

`chunk_assembler.py:861`：

```python
f.truncate(bytes_already_written)
```

逻辑上是安全的（term N 损坏 → N 之后数据不可信），但缺少显式保护。未来维护者可能误解为"截断丢数据"。

### 修改方案

在 truncate 前添加不变量断言 + 注释：

```python
# 顺序写入不变量：若 term N 损坏，则 N 之后的数据不可信，
# 截断到 N 之前的偏移是安全的（丢弃不可信数据，重写恢复）
assert bytes_already_written <= existing_size, (
    f"[ChunkAssembler] truncate 将丢数据: {bytes_already_written} > {existing_size}"
)
f.truncate(bytes_already_written)
```
