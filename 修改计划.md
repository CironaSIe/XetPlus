# 修改计划：Verify + Repair 模式（v2 设计）

---

## 核心原理

### 信任链设计

```
外部锚点（服务器提供）:
  HF Hub resolve API → X-Linked-ETag → 文件级 SHA256
                                             ↓
第一层验证（真正的校验）:
  下载完成后算文件 SHA256 ←→ 比对期望值
        ↓                        ↓
     匹配 ✅              不匹配 ❌
     清理 checkpoint       保留 checkpoint
     报告成功               进入第二层诊断
                                ↓
第二层诊断（定位工具）:
  读取 checkpoint 中 per-term SHA256
  逐 term 重算比对 → 找出损坏的 term
                                ↓
修复:
  只重新下载损坏 term 对应的 xorb
  解压 → 覆写文件损坏部分 → 重新验证
```

### 为什么 per-term SHA256 不能作为第一层？

因为因果问题：
```
下载时服务器返回错误数据 → segment 内容就是错的
                             ↓
SHA256(segment) 算出一个错 hash ← 存入 checkpoint
                             ↓
后续 verify 读文件重算 → 还是这个错 hash → 比对通过
                            （但数据其实错了！）
```

**per-term SHA256 只能检测"正确数据被写坏"的情况，不能检测"数据本身是错的"。**

- 能检测的：竞态写入覆盖（xet-core #604）、bit rot、文件截断、内存错误
- 不能检测的：服务器返回错误、网络传输静默损坏

### 文件级 SHA256 为什么是可信锚点？

- HF Hub resolve API 返回的 `X-Linked-ETag` 是对应 Git LFS 指针文件的 SHA256
- 这个 hash 是上传时算的，存储到 Git 历史中，不可篡改
- 只要最终文件 SHA256 匹配这个值，文件一定正确

---

## per_term_hashes 大小评估
- **实测**: 单条 TermHashRecord JSON ≈ 238B (indented) / 217B (compact)
- **5492 terms**: 约 1.3 MB (indented)
- 对于 281GB 文件来说完全可接受

## 设计方案

### 1. 数据结构变更

**`ReconstructionCheckpoint` (`types.py`)**: v5
- 新增 `expected_sha256: str` — 服务器提供的文件级 SHA256 (v4)
- 新增 `confirmed_bytes: int` — 上次保存时确认已写入磁盘的字节数 (v5)

**侧车文件 `.xet_verify`**: v2
- 新增 `version: 2`
- 保留 `file_hash`、`expected_sha256`、`actual_sha256`

### 2. FileVerifier 重构

```
verify(diagnose=False):
  ├─ 获取 expected_sha256（参数 > checkpoint > 侧车文件）
  ├─ 文件级 SHA256 校验
  │   ├─ 有锚点且匹配 → ✅ 返回 (healthy=True)
  │   ├─ 有锚点但不匹配 → ❌ 记录 mismatch
  │   └─ 无锚点 → ⚠️ 跳过文件级
  └─ 如果 diagnose=True 且文件级失败:
      └─ per-term 诊断 → 记录损坏 term

repair(report=None):
  ├─ 先 verify(diagnose=True)
  ├─ 文件级通过 → ✅ 直接返回
  ├─ 文件级失败 + 无损坏 term → ⚠️ 需要全量重下
  └─ 有损坏 term → 逐 xorb 修复 → 修复后重新校验
```

### 3. CheckpointManager 变更

- `_save_unsafe()` 改用 `checkpoint.to_dict()` 统一序列化
- `save()` 也改用 `_save_unsafe()`（原来有遗漏的字段）
- 新增 `set_expected_sha256(file_hash, sha256)` 方法
- `_load_unsafe()` 新增 `expected_sha256` 反序列化
- `mark_term_completed()` 新增 `confirmed_bytes` 参数

### 4. 续传校验（新增）

**续传逻辑（`chunk_assembler.py:583-688`）:**
- 优先使用 `checkpoint.confirmed_bytes`（直接记录的实际写入量）
- 回退到从 terms 累加（兼容旧版）
- 与 `.part` 文件实际大小对比做三重确认

**三种场景:**
1. 文件大小 ≥ confirmed_bytes → 安全，数据没丢，正常续传
2. 文件介于 confirmed_bytes 与期望值之间 → 可能 OS 未 flush，按实际大小估算
3. 文件 < confirmed_bytes → 数据实际丢失，回退重新下载

### 5. FileReconstructor 变更

- SHA256 失败时调用 `checkpoint_manager.set_expected_sha256()`
- 侧车文件升级到 v2

### 6. CLI 变更

```
xet verify <file>                   文件级 SHA256 校验
xet verify <file> --diagnose        文件级失败后 per-term 诊断
xet verify <file> --repair          诊断 + 修复
```

## 验证场景矩阵

| 场景 | 文件级 SHA256 | per-term 诊断 | 建议操作 |
|------|--------------|--------------|----------|
| 文件完好 | ✅ 通过 | — | 无 |
| 写入时损坏（竞态） | ❌ 不匹配 | ❌ 有损坏 term | `--repair` 修复 |
| 服务器返回错误数据 | ❌ 不匹配 | ✅ 全一致 | 重新下载 |
| 磁盘 bit rot（部分 term） | ❌ 不匹配 | ❌ 有损坏 term | `--repair` 修复 |
| 文件被截断 | ❌ 不匹配 | ⚠️ 检测到截断 | 重新下载 |
| 无 checkpoint | ❌ 不匹配 | 无数据 | 重新下载 |

---

## 边界情况

- **无 checkpoint + 无侧车文件** → 提示用户无法验证
- **下载完成后 checkpoint 被清理** → 文件级 SHA256 仍然可用（如果侧车文件还在）
- **修复后文件级 SHA256 仍然不匹配** → 报告修复失败，建议全量重下
- **修复中断** → 重新运行 `--repair` 跳过已修复的 term
