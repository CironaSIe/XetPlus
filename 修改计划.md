## 一、问题背景

### 当前内存模型（重）

```
下载 xorb 压缩数据 (16MB)
  → _decompress_single_xorb() 全量 LZ4 解压所有 chunk
    → XorbBlockData.data = 64MB+ 解压后全量字节
      → 存入 _xorb_cache: OrderedDict
        → 每个term提取: xorb_data.data[start_byte:end_byte]  (切片复制 ~4MB)
          → 写入文件后释放整个 XorbBlockData
```

**内存热点**：
- 单个 xorb 在缓存中占用 **64MB+**（解压后）
- 预取机制可能同时持有 **3-4 个 xorb** = **192-256MB**
- L1 修复前自适应水位线可达 **840MB**
- L1 修复后硬上限 **200MB**，但每个 xorb 仍然占 64MB，只能缓存 3 个

### 目标内存模型（轻）

```
下载 xorb 压缩数据 (16MB)
  → StreamingXorbAccessor: 仅扫描 chunk header 索引 (~8KB)
    → 保留原始压缩数据 (16MB) + 索引 (~8KB) = ~16MB
      → 存入 _xorb_cache
        → 每个term按需解压: accessor.extract_range(start, end)  (仅解压涉及的 1-2 chunks ≈ 128KB)
          → 写入文件后释放 accessor (16MB vs 64MB)
```

**预期收益**：
| 维度 | 当前 | 流式后 |
|------|------|--------|
| 单 xorb 缓存占用 | 64MB (解压后) | 16MB (压缩原始) |
| 同时缓存 xorb 数量 (200MB上限) | 3 个 | 12 个 |
| 提取 term 数据时分配 | 切片 64MB→复制 4MB | 解压 1-2 chunks ≈ 128KB |
| 峰值内存 (281GB 文件) | ~200MB (L1修复后) | ~64-96MB |

## 二、技术可行性

### Xorb 二进制格式天然支持随机访问

来源：[xorb_deserializer.py](xet/storage/xorb_deserializer.py) 第 222-237 行

```
┌─────────────────────────────────────────────────────────────┐
│  Chunk #0           │  Chunk #1           │  Chunk #N...    │
│ ┌──────┬──────────┐ │ ┌──────┬──────────┐ │                │
│ │Header│ Compressed│ │ │Header│ Compressed│ │   ...          │
│ │ 8B   │ Data      │ │ │ 8B   │ Data      │ │                │
│ └──────┴──────────┘ │ └──────┴──────────┘ │                │
└─────────────────────────────────────────────────────────────┘

Header (固定8字节):
  version(1B) | compressed_len(3B u24 LE) | scheme(1B) | uncompressed_len(3B u24 LE)
```

**关键特性保证流式可行**：

| 特性 | 值 | 流式友好度 |
|------|-----|-----------|
| 每个 chunk 独立压缩 | LZ4 / None / ByteGrouping4LZ4 | 完美 — 无跨chunk依赖 |
| Header 固定 8 字节 | 包含精确的 compressed_len | 可以 O(1) 定位任意 chunk |
| chunk 顺序排列 | offset 可通过累加计算 | 单次扫描即可建索引 |
| 解压函数独立 | `_decompress(data, scheme, size)` | 可按需调用单个 chunk |

## 三、StreamingXorbAccessor 设计

### 3.1 类定义位置

新增于 [xorb_deserializer.py](xet/storage/xorb_deserializer.py)，与现有 `XorbBlockData` 同文件。

### 3.2 接口设计

```python
class StreamingXorbAccessor:
    """流式 xorb 访问器 — 不一次性全量解压。

    只扫描 chunk header 建索引 (~8KB)，保留压缩原始数据。
    按需解压单个 chunk 或范围。

    对比 XorbBlockData:
    - XorbBlockData:   64MB 全量解压数据 + offsets
    - StreamingXorb:   16MB 压缩原始 + 8KB 索引 + 按需解压
    """

    def __init__(self, raw_bytes: bytes, chunk_offsets=None):
        """
        Args:
            raw_bytes: xorb 原始字节数据（压缩格式）
            chunk_offsets: 外部提供的 [(global_chunk_id, virtual_byte_offset), ...]
                           多段合并场景传入；单段时为 None（自动从 header 扫描生成）
        """

    # ── 兼容 XorbBlockData 的接口 ──

    @property
    def chunk_offsets(self) -> List[Tuple[int, int]]:
        """[(global_chunk_id, virtual_byte_offset), ...] — 与 XorbBlockData 格式完全一致"""

    def data_size(self) -> int:
        """虚拟数据总大小（所有 chunk uncompressed_len 之和），用于边界检查"""

    def memory_footprint(self) -> int:
        """实际内存占用 = len(raw_bytes)，用于水位线记账"""

    def get_chunk_byte_indices(self) -> List[int]:
        """兼容接口 — 返回 [0, off1, off2, ..., total]"""

    # ── 流式核心方法 ──

    def extract_range(self, start_byte: int, end_byte: int) -> bytes:
        """按需提取字节范围（只解压涉及的 chunks）。

        替代 xorb_data.data[start:end] 的核心方法。
        算法：
          1. 二分查找 start/end 覆盖哪些 chunk
          2. 仅解压这些 chunk（通常 1-2 个）
          3. 拼接后切片返回
        """

    def decompress_all(self) -> 'XorbBlockData':
        """全量解压（懒加载，缓存结果）。

        仅在需要完整数据的场景调用：
        - chunk 缓存写入 (put_xorb_decompressed)
        - xorb_adapter.py 兼容路径

        结果缓存避免重复解压。
        """
```

### 3.3 两种构造模式

**模式 A：压缩数据（主流径）**
```python
# 下载得到的原始 xorb 数据（来自 HTTP/CAS/磁盘缓存）
accessor = StreamingXorbAccessor(compressed_data_from_download)
# 内部自动扫描 header → 构建 _index
```

**模式 B：已解压数据（chunk 缓存命中路径）**
```python
# 从 chunk 磁盘缓存加载的数据已经是解压后的
decompressed_data, byte_indices = cache_adapter.get_xorb_decompressed(...)
accessor = StreamingXorbAccessor(
    raw_bytes=decompressed_data,   # 已解压的数据作为 "raw"
    chunk_offsets=constructed_offsets  # 手动提供 offsets
)
# 标记为 "pre-decompressed" 模式，extract_range 直接切片不解压
```

`_index` 内部结构：
```python
@dataclass
class _ChunkEntry:
    global_chunk_id: int       # 全局 chunk 编号
    virtual_offset: int        # 在虚拟数据空间中的偏移（= 前面所有 chunk 的 uncompressed_len 之和）
    data_offset: int           # 在 raw_bytes 中的偏移（header 后面）
    compressed_len: int        # 压缩数据长度
    comp_scheme: int           # 压缩方案 (0/1/2)
    uncompressed_len: int      # 解压后长度
```

`extract_range` 算法复杂度：
- _index 是有序的（按 virtual_offset 排序）
- 使用二分查找定位 start/end 对应的 chunk 范围
- 通常一个 term 只涉及 1-2 个 chunk → 只解压 1-2 次
- 最坏情况（term 跨很多 chunk）：线性扫描 + 多次解压，但这种情况极少

| 子项 | 行号 | 当前代码 | 新代码 | 副作用 |

**L636 改动的详细副作用分析**:

当前行为：
```python
segment = xorb_data.data[start_byte:end_byte]
# 1. 从 64MB 的 bytes 对象中切片
# 2. Python bytes 切片产生新对象（copy-on-write，但在 CPython 中可能触发 memcpy）
# 3. 典型 segment 大小: 4-64 KB
```

新行为：
```python
segment = xorb_data.extract_range(start_byte, end_byte)
# 1. 二分查找 _index 确定涉及的 chunks（通常 1-2 个）
# 2. 对每个涉及的 chunk 调用 _decompress()（LZ4 解压 ~64KB）
# 3. 拼接后切片返回
# 4. 典型开销: 1 次 LZ4 解压 ≈ 0.1-0.5ms
```

**性能对比**:
| 操作 | 当前 | 流式 |
|------|------|------|
| 内存分配 | 切片 64MB bytes → copy ~4KB | 解压 1 chunk → alloc ~64KB |
| CPU 开销 | 几乎为零（bytes 切片） | LZ4 解压 ~0.1-0.5ms/chunk |
| 缓存友好性 | 差（访问 64MB 中的 4KB） | 好（只操作需要的 64KB） |

**结论**: 每次 extract_range 多花 ~0.5ms LZ4 解压，但省去了持有 64MB 全量数据的代价。对于 281GB 文件的数千个 terms 来说，总额外 CPU 开销约 1-2 秒，但内存收益巨大。

**改动点 2C**: 并行写入模式数据提取 (L786-820)

与 2B 完全对称的模式，同样的改动：
- L812: `len(xorb_data.data)` → `xorb_data.data_size()`
- L814: `len(xorb_data.data)` → `xorb_data.data_size()` (越界检查)
- **L820**: `xorb_data.data[start:end]` → `xorb_data.extract_range(start, end)`

**改动点 2D**: 内存水位线记账 (多处 `len(x.data)`)

| 位置 | 行号 | 当前 | 新代码 | 数值变化影响 |
|------|------|------|--------|-------------|
| prefetch check (seq) | L589 | `sum(len(x.data) for ...)` | `sum(x.memory_footprint() for ...)` | 64MB→16MB per xorb → 水位线认为"缓存更小"→ 触发更多预取 ✅ |
| release logging (seq) | L671 | `len(self._xorb_cache[...].data)` | `.memory_footprint()` | 日志数字变小，无功能影响 |
| periodic log (seq) | L680 | `sum(len(x.data) for ...)` | 同上 | 日志数字变小 |
| prefetch check (par) | L778 | 同 L589 | 同上 | 同上 |
| release logging (par) | L860 | 同 L671 | 同上 | 同上 |
| periodic log (par) | L869 | 同 L680 | 同上 | 同上 |
| batched mode | L253, L270, L310 | `len(xorb_data.data)` | `.memory_footprint()` | batched 已弃用，低优先级 |

**关键副作用 — 水位线行为变化**:

当 `len(.data)` (64MB) 变为 `.memory_footprint()` (16MB) 后：
- `current_cache_bytes` 的值变为原来的 ~25%
- `current_cache_bytes < low_watermark` 更容易为 True → **更积极地触发预取**
- 这正是我们想要的：因为实际内存占用确实更低了，可以安全地缓存更多 xorb

**改动点 2E**: 缓存写入路径 (`xorb_data.data` 传给 adapter)

| 位置 | 行号 | 当前代码 | 问题 | 解决方案 |
|------|------|---------|------|---------|
| disk cache load | L95-96 | `cache_adapter.put_xorb_decompressed(..., xorb_data.data, ...)` | StreamingXorbAccessor 没有 `.data` 属性 | 改为 `xorb_data.decompress_all().data` |

**本文件风险总评**: **MEDIUM** — 核心热路径改动，但接口设计为兼容模式，每处改动都有明确的等价映射。

### 7.3 回滚方案

1. `_decompress_single_xorb` 恢复返回 `XorbBlockData`
2. `extract_range()` 调用恢复为 `.data[start:end]`
3. `.memory_footprint()` 恢复为 `len(.data)`
4. fallocate 恢复

## 八、风险矩阵汇总

| # | 改动 | 风险等级 | 影响 | 缓解措施 |
|---|------|---------|------|---------|
| 1 | 新增 StreamingXorbAccessor | LOW | 无 | 纯新增，不影响现有代码 |
| 2 | _decompress_single_xorb 返回类型 | MEDIUM | 所有缓存存储位置 | 统一接口，duck typing |
| 3 | extract_range 替换 .data[:] | MEDIUM-HIGH | 每个 term 的数据提取 | 核心算法需仔细测试 |
| 4 | memory_footprint 替换 len(.data) | LOW | 水位线预取行为 | 值变小→更积极预取（良性） |
| 5 | 缓存写入改用 .decompress_all() | LOW | 磁盘缓存写入 | 非热路径，一次性的 |
| 6 | chunk 缓存预解压模式 | LOW-MED | 缓存恢复路径 | 需确保双模式分支正确 |
| 7 | fallocate 移除 | LOW | 启动速度 | GlobalWriter 自带文件创建 |
