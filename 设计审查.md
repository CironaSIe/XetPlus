# 设计审查：内存 / 下载速度 / 写入速度瓶颈分析

> 审查基于当前代码库，涵盖三种下载模式的全部数据流。

---

## P0 — 必须修复

### 1. 模式③ 全段内存驻留

**文件**: `segmented_reconstructor.py`

`_download_and_assemble_segment()` 把整个段读进内存：
```python
self._stream_assembly_to_temp(seg, recon, seg_temp)  # 写 .tmp 文件
return seg_temp.read_bytes()  # ← 256MB bytes 对象
```

然后这个 `bytes` 通过 write queue 传递给 writer。队列最大 `parallel_segments * 2` 项。

**并发 4、段 256MB 时的最坏内存**：
| 来源 | 内存 |
|------|------|
| 4 个 worker 各持有 1 个段 | 1 GB |
| Write queue 积压 4-8 个段 | 1-2 GB |
| 每个 worker 的 ChunkAssembler xorb 缓存 | ~200 MB × 4 |
| **合计** | **~2.5-3.5 GB** |

**方案**：writer 直接从临时文件读取，worker 只需传递 seg_index 和文件路径：
```python
# worker：下载到临时文件后通知 writer
self._write_queue.put((seg.start, seg.index, seg_temp))
# writer：从临时文件分段读取并写入
with open(seg_temp, 'rb') as src:
    shutil.copyfileobj(src, f)
seg_temp.unlink()
```

这样 worker 不持有段数据，队列项大小从 256MB 降至 ~100 bytes。

### 2. 模式③ Writer 逐个 fsync

**文件**: `segmented_reconstructor.py:_writer_worker`

每写一个段就做 `os.fsync(f.fileno())`。对 256MB 段，fsync 耗时 10-500ms，期间 writer 阻塞。

**方案**：积累多个段写入后做批量 fsync（类似 `GlobalWriter` 的 batch 模式）：
```python
# 每 N 个段或每 M 秒做一次 fsync
write_count = 0
...
if write_count % 5 == 0:  # 每 5 个段 fsync
    os.fsync(f.fileno())
```
或在文件末尾一次性 fsync。

### 3. ChunkAssembler 多段 xorb 内存计账不对称

**文件**: `chunk_assembler.py` / `chunk_assembler_helpers.py`

单段 xorb：`StreamingXorbAccessor` 持有压缩数据，`memory_footprint()` 返回压缩大小。
多段 xorb：LZ4 全解压后在 `bytearray` 之上封装 `StreamingXorbAccessor(raw_bytes=bytes(...))`，`memory_footprint()` 返回**解压后**大小——通常大 2-4 倍。

水印系统按 `memory_footprint()` 做淘汰判断，导致多段 xorb 占比虚高、过早驱逐单段 xorb。

**方案**：`memory_footprint()` 统一返回压缩大小，或在 `StreamingXorbAccessor` 上区分两种模式。

## P1 — 重要优化

### 4. ChunkAssembler 顺序写无 fsync

**文件**: `chunk_assembler.py:_assemble_with_sequential_write`

所有 `f.write()` 后只有 `f.close()`（隐式 flush），无显式 fsync。如果进程在写完后、rename 前崩溃，`.part` 文件内容可能不完整。

**方案**：按一定间隔（每 50 个 term 或每 10MB）做一次 `os.fsync()`，在性能和安全性之间平衡。

### 5. 模式② 每段二次 I/O

**文件**: `segmented_reconstructor.py:_process_segment`

每段先写 `.tmp` 文件（`_stream_assembly_to_temp`），再 `copyfileobj` 到最终文件。对于顺序模式，临时文件是多余的。

**方案**: 顺序模式下让 ChunkAssembler 直接写入最终文件的正确偏移，跳过 `.tmp`。

### 6. 每段 Reconstruction API 调用

**文件**: `segmented_reconstructor.py`

每个段首次处理都要调 `get_segment_reconstruction()`，单个调用可能耗时 ~2 分钟。并行模式下 4 个段同时调，串行模式下按顺序调。

**方案**: 支持在下载当前段的同时预取下一段的 reconstruction。或利用 `recon.json` 缓存跨运行复用。

### 7. 预取在同步下载后才触发

**文件**: `chunk_assembler_helpers.py:_ensure_xorb_ready`

`_ensure_xorb_ready()` 先**同步下载**当前 xorb，然后才检查水印是否触发预取。如果当前 xorb 只有 1-2 个 term，预取来不及完成就被消费。

**方案**: 在同步下载前检查水印并触发预取：
```python
def _ensure_xorb_ready(self, xorb_hash, current_term_idx):
    if xorb_hash in self._xorb_cache:
        return
    self._prefetch_upcoming_xorbs(current_term_idx)  # ← 提前预取
    ...
```

## P2 — 次要优化

### 8. Cache 写触发全量 LZ4 解压

每次 cache-miss 下载 xorb 后，`put_xorb_decompressed()` 调用 `xorb_data.decompress_all()` 解压全部 chunk 写入磁盘缓存。即使文件只需要其中一部分 chunk。

**方案**: 递延缓存写入：在 xorb 被从 `_xorb_cache` 驱逐时才写入磁盘缓存，而非下载后立即写入。

### 9. Write queue 按项数而非字节数限流

`queue.Queue(maxsize=parallel_segments * 2)` 对 256MB 段意味着最大 2GB 积压。

**方案**: 增加字节级信号量 `_queued_bytes`，每个 worker `put` 前 acquire、writer 处理完 release。上限设为 `max_memory_mb` 的 50%。

## 数据流总览

```
模式① FileReconstructor:
  get_reconstruction → ChunkAssembler
    → _ensure_xorb_ready(同步下载) → 提取 term → f.write(term)
    → [无 fsync] → f.close() → rename .part

模式② SegmentedReconstructor._reconstruct_sequential:
  get_segment_reconstruction → _stream_assembly_to_temp → .tmp 文件
    → copyfileobj(.tmp → output) → fsync → mark_completed

模式③ SegmentedReconstructor._reconstruct_parallel:
  [worker 1..N] get_segment_reconstruction → _stream_assembly_to_temp → .tmp
    → seg_temp.read_bytes() → queue.put(offset, data, seg_index)
  [writer] queue.get → f.write → fsync → mark_completed
```

## 锁竞争评估

| 锁 | 竞争程度 | 说明 |
|----|---------|------|
| `CheckpointManager._lock` | 中等 | 每 checkpoint_interval(10) term 获取一次做 JSON I/O |
| `SegmentedReconstructor._lock` | 低 | 仅 writer 线程访问 |
| `CASClient._recon_cache_lock` | 低-中 | 短暂，保护 dict 操作 |
| `AdaptiveConcurrencyController._lock` | 低-中 | 每成功/失败更新 |
| `ProgressTracker._lock` | 低 | 快速计数器加锁 |
