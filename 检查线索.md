# 流式 XET 校验集成检查线索

## 目标

在现有下载管道中注入 XET hash chain 校验（chunk_hash → xorb_hash → file_hash），支持带数据正确性检验的流式下载。

> **2026-06-29 更新**：跨文件补全已确认不可行（API 限制）。xorb_hash 仅在数据完整时验证，file_hash 为主校验。

## 当前校验缺口

| 阶段 | 现有校验 | XET 链校验 | 可验性 |
|------|---------|-----------|--------|
| chunk 级解压后 | 无 | `compute_data_hash_bytes(uncompressed)` | ✅ 可加，但对比无锚点（footer 被剥离），仅供上游积累 |
| xorb 完成处理时 | 无 | `compute_xorb_hash_from_hashes()` vs `term.hash` | ✅ 可加，需该 xorb 所有 chunk（**默认激活，跨文件补全**） |
| 文件组装完成 | SHA256（来自 X-Linked-ETag 或 CLI 参数） | `compute_file_hash_from_hashes()` vs `files` key | ✅ 可加，端到端验证，最可靠 |
| 续传时文件尾部校验 | per-term SHA256 回溯 | — | ✅ 已有，可保留 |

## 插入点分析

### Phase 0 — 预构建 repo_xorb_map（新增）

**位置**：`download.py` — 在下载任何文件之前

**流程**：
```
for each file in repo:
  获取 reconstruction 信息
  提取每个 term 的 (xorb_hash, chunk_range, file_offset)

构建反向索引:
  repo_xorb_map[xorb_hash] = {
    "total_chunks": 100,
    "files": {
      "file_A": [{"term_idx": 0, "chunk_range": (0, 10), "file_offset": ...}],
      ...
    }
  }

缓存到 .xet/cache/repo_xorb_map_{commit_sha}.json
```

**内存开销**：假设 20000 个 xorb，每个 ~1.3 KB → ~26 MB（可接受）

**进度条**：
```
正在构建仓库 xorb 映射表...
████████░░░░░░░░░░░░ 42% (420/1000 文件)
```

> **2026-06-29 实测**：BATCH API 只返回当前文件引用的 chunk，无法获取同一仓库其他文件中的同 xorb chunk。`repo_xorb_map` 用于覆盖度分析和确认 gap 不影响文件完整性，不用于运行时补全。

### Phase 1 — 主插入点：`xet/pipeline/chunk_assembler.py`

**位置**：`_assemble_with_sequential_write()` 的 term 处理循环（约 line 891-1005）

**流程**：
```
for each term:
  1. ensure_xorb_ready(term.hash)
  2. decompress chunk → segment
  3. compute_data_hash_bytes(uncompressed_chunk) ← ← ← 注入点①
     → accumulate (chunk_id, chunk_hash_hex, chunk_size) per xorb_hash
  4. 检查 xorb 是否完整（跨文件补全）:
       if len(fetched_chunk_ids) < repo_xorb_map[term.hash]["total_chunks"]:
         # 查表：缺失的 chunk 在哪些文件
         for other_file, other_term in repo_xorb_map[term.hash]["files"]:
           下载 other_file 的 other_term 的 xorb 数据
           解压缺失的 chunk → 积累到 xorb_accumulator[term.hash]
       
       # 现在 xorb 完整了
       computed_xorb_hash = xorb_hash_from_hashes(xorb_accumulator[term.hash])
       if computed_xorb_hash != term.hash:
         raise ReconstructionError(f"xorb {term.hash[:16]} 验证失败")
       ← ← ← 注入点②
  5. write segment to file (已有)
  6. compute per-term SHA256 (已有)
  7. if last term of file:
       compute_file_hash_from_hashes(all_collected) vs files[file_hash]
       ← ← ← 注入点③
```

**内存开销**：
- xorb_accumulator：每 chunk 68 bytes (64B hex + 4B size)。4K chunks ≈ 272 KB（完全流式）
- file_hash_accumulator：同上

**进度条**：
```
下载 large_model.bin...
████████████████░░░░ 80% (80/100 term)
补全 xorb_A... ████████████████████ 100% (50 MB / 50 MB)
```

### Phase 3 — 分段模式：`xet/pipeline/segmented_reconstructor.py`

分段模式按文件字节范围分段，每段独立请求 reconstruction。

**xorb_hash 验证**：一个 xorb 的 chunk 通常跨越多个段（实测 xorb_00 的 chunk 分布在 3 个 term 中），且跨文件共享。xorb_hash 验证需要**跨段积累 + 跨文件补全**。分段模式下：
- `ChunkAssembler` 不变：每 term 积累 `(chunk_hash_hex, size)`
- `SegmentedReconstructor` 持有全局的 `xorb_accumulator` 和 `file_hash_accumulator`
- 传给每个 segment 的 `ChunkAssembler`，由它积累
- 所有段完成后，`SegmentedReconstructor` 做 xorb_hash 和 file_hash 校验

**注意**：段级 checkpoint 导致文件分多次重建请求，**每个请求的 `files` key 可能不同**（file_hash 是全文件的，段级 API 通常返回的是子集的 file_hash）。不能用段级 entry 验证全文件 file_hash。

## 跨文件 xorb 共享问题 — 跨文件补全不可行

**现实**：同一个 xorb 的所有 chunk 很可能分布在同一仓库里不同的 XET 文件中。

**2026-06-29 关键发现**：BATCH API 的 `fetch_info` 只返回当前请求文件引用的 xorb 数据段，无法获取完整 xorb。

**调整后策略**：
1. **file_hash 是主校验** — 不受 xorb 完整度影响，端到端验证文件完整性
2. **xorb_hash 降级为可选** — 仅在当前文件恰好覆盖全部 chunk 时验证
3. **repo_xorb_map 用于覆盖度分析** — 确认 gap 不影响文件完整性，不用于运行时补全

**跨文件去重**仍然有意义：同一仓库下载多个文件时，共享 xorb 只下载一次，减少传输量。

## XET Hash 链 API 一览（`xet.storage.merkle_hash`）

| 函数 | 输入 | 输出 | 用途 |
|------|------|------|------|
| `compute_data_hash_bytes(data)` | chunk 原始数据 | bytes(32) | 每 chunk 解压后立即调用 |
| `compute_xorb_hash_from_hashes(chunks_hex)` | `[(hash_hex, size)]` | hex(64) | xorb 完成时对比 term.hash |
| `compute_file_hash_from_hashes(chunks_hex)` | `[(hash_hex, size)]` | hex(64) | 文件完成时对比 files key |
| `hash_to_hex(hash_bytes)` | bytes(32) | hex(64) | MerkleHash → 64 hex chars |
| `lz4_frame_decompress(data)` | LZ4 frame 压缩数据 | bytes | scheme 1 解压 |
| `bg4_lz4_decompress(data)` | BG4+LZ4 压缩数据 | bytes | scheme 2 解压 |
